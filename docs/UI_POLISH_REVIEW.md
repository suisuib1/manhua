# 小程序 UI 修复审查方案

## 1. 当前 UI 总体问题

本轮审查基于已完成首页、创建章节页、PRD、MVP 范围、UI 页面映射和 `docs/ui/` 原型图。当前最大问题不是单个颜色或间距，而是剩余页面没有形成和首页一致的视觉系统。

主要问题如下：

- 首页和创建页已经形成“暖白背景 + 手账卡片 + Q 版插画 + 粉色主按钮”的方向，但后续页面多数只是复制了相似卡片，缺少页面主视觉。
- diary、character、generating、chapter-detail、share、mine、quota-empty 的结构过于模板化：标题、卡片、卡片、按钮，页面之间差异小，情绪记忆点弱。
- 多数页面没有继承 `docs/ui/` 的核心构图：原型图通常先给出大插画或大手账容器，再承载输入、状态或漫画内容；当前实现常把所有信息切成多个平级卡片。
- 视觉层级不稳定：页面标题、卡片标题、正文、辅助说明、按钮文字的字号和粗细虽然接近，但没有明确的“主视觉标题 / 功能区标题 / 提示文案”层级。
- 卡片系统不统一：圆角、边框、虚线、阴影、内边距、背景色在不同页面重复但略有差异，导致像不同模板拼接。
- 信息密度在部分页面偏高，尤其 diary、character、chapter-detail、share：用户在 375px 宽度下会连续看到大量文本和卡片，缺少呼吸感。
- CSS 画出的占位插画质感弱，容易显得像工程占位，而不是 Q 版漫画氛围。
- 部分页面有“占位”“静态展示”等开发态文案，容易破坏产品感；这些可以留在 toast 或注释，但不建议展示在主界面。
- 当前代码读取时存在中文乱码显示迹象，需要在后续修复中确认文件编码和页面真实展示效果，避免 WXML 文案出现不可读字符。
- 隐私和分享边界基本有覆盖，但 share 和 chapter-detail 需要更强地突出“只展示摘要和漫画，不展示原文”，避免淹没在普通说明卡里。

## 2. 统一视觉规范

后续修复应先统一规则，再逐页改造。建议以首页和创建页为基准，但更靠近 `docs/ui/` 的手账质感。

### 2.1 页面骨架

- 页面宽度按 375px 小程序视口设计，内容左右边距统一为 `28rpx`。
- 页面背景统一为 `#fffaf5`，允许轻微暖色径向光晕，但每页最多 2 个背景光斑。
- 每页只保留一个明确主视觉区：
  - diary：大日记本输入卡。
  - character：角色档案封面卡。
  - generating：大生成插画 / 状态舞台。
  - chapter-detail：漫画阅读区。
  - share：私密分享预览卡。
  - mine：用户与漫画书信息卡。
  - quota-empty：空状态插画卡。
- 页面主视觉区之后再放辅助信息，避免 4 个以上平级卡片连续堆叠。

### 2.2 卡片规范

- 主卡片：圆角 `34-44rpx`，边框 `2-3rpx`，可使用虚线内框，阴影较轻。
- 次级卡片：圆角 `24-30rpx`，边框 `2rpx`，少用阴影。
- 提示条：高度不宜过高，使用浅黄或浅粉底，图标 + 一行半文案。
- 禁止每个页面都使用完全一样的 `.form-card` 堆叠；应按页面任务设计主容器。

### 2.3 色彩规范

- 主背景：`#fffaf5`。
- 主文字：`#4a2b22` / `#50332a`。
- 次级文字：`#75574c` / `#8d7166`。
- 主按钮：粉色渐变 `#ff96a7` 到 `#f77187`。
- 辅助按钮：暖白底 + 浅棕边框，不使用高饱和颜色。
- 提示区：浅黄 `#fff7e8` 或浅粉 `#fff1f4`。
- 避免新增大面积紫色、绿色、蓝色卡片；这些颜色只用于极小点缀。

### 2.4 字体层级

- 页面标题：`38-42rpx`，`font-weight: 800`。
- 主视觉标题：`36-44rpx`，根据页面情绪可更突出。
- 卡片标题：`30-32rpx`。
- 正文：`25-28rpx`，行高 `1.55-1.7`。
- 辅助说明：`22-24rpx`。
- 主按钮文字：`31-34rpx`。

### 2.5 按钮规范

- 每页最多一个主按钮。
- 主按钮宽度 `600-620rpx`，高度 `96rpx`，适合单手点击。
- 次级操作不要和主按钮同权重；chapter-detail 的删除按钮必须弱化。
- “占位”字样不出现在主要按钮上，改为正常产品文案，点击后 toast 再说明当前为静态演示。

### 2.6 插画与装饰

- 优先使用已有素材或更接近原型图的轻插画区，不要每页重新画一套简陋 CSS 人物。
- 装饰只保留少量星星、贴纸、纸夹、虚线内框，避免页面碎。
- 漫画图占位要像“漫画结果”，不要像普通色块或工程 grid。

## 3. 页面级问题清单

### 3.1 首页 index

状态：相对最好，可作为基准。

问题：

- 顶部和漫画书卡片已经有风格，但真实文件读取时出现中文乱码迹象，需要确认开发者工具中的实际展示。
- 首页已有素材接入，视觉完成度高于其他页。
- 最近章节横向卡片信息略多，但仍可接受。

修复建议：

- 本轮不建议优先改首页。
- 后续只做微调：确认编码、减少乱码符号、统一文案。

### 3.2 创建章节页 create

状态：较好，可作为表单页基准。

问题：

- 页面结构比首页更接近表单，但整体还保持手账感。
- 部分符号图标在读取时出现乱码，需确认真实渲染。
- 卡片层级较多，但任务本身是配置表单，可接受。

修复建议：

- 本轮不建议优先改创建页。
- 可作为 diary / character 的表单样式参考，但不要简单复制卡片堆叠。

### 3.3 日记输入页 diary

混乱程度：高。建议第一批修。

问题：

- 与 `03-diary-input.png` 差距较大。原型的核心是“大日记本 + 上传照片区域 + 生成前检查”，当前变成多个普通卡片。
- 当前页面先展示“当前章节摘要”，抢占了日记输入的主任务焦点。
- 日记正文卡片太普通，没有手账本装订、虚线纸张、右上字数的视觉记忆点。
- 照片区域使用纯色占位，缺少照片感和 Q 版手账氛围。
- 隐私提示、角色提示、章节摘要分散为多张卡，导致页面长且碎。
- “本地占位预览”等开发态文案不应作为产品主文案。

修复方向：

- 重排为：顶部标题与气泡说明 -> 大日记本输入卡 -> 隐私提示条 -> 上传照片主卡 -> 生成前检查卡 -> 底部主按钮。
- 将“当前章节配置摘要”压缩为日记卡上方或生成前检查卡的一行信息。
- 照片占位改为 2:1 或 1:1 圆角缩略图，保持 3 列，但视觉上更像照片贴纸。
- 隐私提示必须保留，但做成一条醒目的浅黄提示条。

### 3.4 角色档案页 character

混乱程度：高。建议第一批修。

问题：

- 与 `04-character-profile.png` 的档案卡布局差距较大。原型是“角色封面 + 多字段档案表”，当前只有头像、昵称、描述和两组标签，信息结构偏薄。
- 当前角色头像 CSS 插画质感较弱，像占位头像，不像角色档案封面。
- 表单字段过少，和 PRD 里的角色档案边界不完全匹配。
- 标签区连续两张卡，缺少统一档案纸张容器。
- 保存按钮前的说明卡较生硬。

修复方向：

- 重排为：顶部标题 -> 角色档案封面大卡 -> 档案表单大卡 -> 参考图/头像占位区域 -> 角色一致性提示条 -> 保存按钮。
- 字段至少覆盖：昵称、身份/关系、年龄段、发型、脸型、常穿服装、标志性配饰、性格关键词。
- 不做真实上传，不做训练，只保留静态/本地选择。

### 3.5 生成等待页 generating

混乱程度：中高。建议第二批修。

问题：

- 与 `05-generating.png` 差距明显。原型是一个大插画舞台，当前是简化 CSS 头像 + 进度条 + 步骤列表。
- 当前步骤列表信息过多，偏后台任务面板，不像治愈等待页。
- 进度百分比很工具化，和“漫画慢慢长出来”的氛围不一致。
- 失败重试入口在等待状态下露出，视觉上像异常优先，容易让用户焦虑。

修复方向：

- 重排为：顶部标题 -> 大插画/手账生成舞台 -> 当前章节提示 -> 三步进度条（排队中/生成中/整理漫画页） -> 等待提示 -> 返回漫画书按钮。
- 可以保留本地 timer，但 UI 上弱化百分比。
- 失败重试入口只作为很弱的文本入口或失败态预留，不在正常等待页主按钮区展示。

### 3.6 章节详情页 chapter-detail

混乱程度：最高。建议第一批修。

问题：

- 当前漫画展示像普通网格色块，不像“漫画阅读页”。
- 页面顶部信息、漫画页、摘要、隐私说明、三个操作按钮全部平铺，阅读体验碎。
- `06-chapter-detail.png` 的重点应是漫画内容，当前漫画内容没有视觉权重。
- 删除按钮文案带“占位”，破坏产品感。
- 隐私说明存在，但没有和漫画阅读区形成清晰边界。

修复方向：

- 重排为：章节头部 -> 漫画阅读主容器 -> 摘要/隐私折叠式提示 -> 操作区。
- 漫画占位应改成纵向大图卡，每张图内部有 3-4 格分镜，比例更接近漫画结果。
- 分享按钮是主操作；重新生成和删除是次级/危险操作，视觉上降低权重。
- 不展示完整日记，只展示摘要和隐私提示。

### 3.7 私密分享页 share

混乱程度：中高。建议第二批修。

问题：

- 当前像内部“创建分享链接”页，不像外部访问者看到的私密分享页。
- link box 展示 `mock://...` 让页面工程感明显。
- 复制链接 / 微信分享按钮属于拥有者操作，但分享页需求更像“被分享者查看章节”；两种场景混在一起。
- 隐私说明是普通 notes 列表，权重不足。

修复方向：

- 明确场景：如果是“用户发起分享页”，则展示链接状态和复制按钮；如果是“访问者分享页”，则展示章节内容和打开小程序按钮。当前 MVP 文档更偏章节私密分享页，建议先按发起分享管理页做低风险版本。
- 移除裸露 mock URL，改成“私密链接已准备好”状态。
- 章节预览要更像漫画卡片，不是两个普通色块。
- 隐私边界放在主卡底部醒目说明。

### 3.8 我的页 mine

混乱程度：中。建议第三批修。

问题：

- 当前结构很常见，但偏通用设置页，缺少“私人漫画书”的情绪。
- 统计卡、免费次数卡、菜单卡都是普通管理后台式卡片。
- 菜单图标是统一渐变块，残留模板感明显。
- 角色档案入口、隐私说明入口、关于产品入口没有明显优先级。

修复方向：

- 重排为：用户漫画书封面卡 -> 免费次数轻卡 -> 功能入口列表。
- 默认漫画书统计要视觉化：章节数、已完成、生成中可以放在一张“书本信息卡”里。
- 菜单图标用不同轻量符号或已有素材，不使用一模一样的渐变方块。
- 不增加会员、支付、充值等入口。

### 3.9 免费次数不足页 quota-empty

混乱程度：中。建议第三批修。

问题：

- 当前基本符合空状态，但 CSS 小猫/月亮较简陋。
- 文案和按钮可用，但视觉冲击弱，缺少原型 `09-Failure.png` 的可爱失败态。
- tips 卡和状态卡分离略显多余。

修复方向：

- 做成单一大空状态卡：插画 -> 标题 -> 温和说明 -> 恢复提示 -> 两个按钮。
- 去掉额外 tips 卡或合并到状态卡内。
- 保持“不引导付费”，不出现购买类入口。

## 4. 页面修复优先级

P0：必须先修，影响主流程观感。

1. `chapter-detail`：漫画结果页最重要，但当前最不像漫画阅读页。
2. `diary`：创建主流程核心页面，和原型差距大。
3. `character`：角色档案是产品差异点，当前模板感重。

P1：第二批修，影响流程完整度。

4. `generating`：等待页需要强情绪主视觉，当前偏任务面板。
5. `share`：隐私分享边界需要更清晰，去掉工程感。

P2：第三批修，做统一收尾。

6. `mine`：当前可用但普通，后续统一风格。
7. `quota-empty`：可用但插画和结构需更精致。
8. `index` / `create`：只做微调和编码确认，不作为重构对象。

## 5. 建议先修哪几个页面

建议下一轮不要继续批量修全部页面。先单独执行一个“小目标”：

首选：`chapter-detail` 页面 UI 修复。

原因：

- 它是漫画生成结果的核心承载页，最能决定产品是否像“日记生成漫画”。
- 当前问题集中且明显，修复收益最高。
- 修好后可沉淀漫画占位、隐私提示、操作按钮规范，供 share 和 home 最近章节复用视觉语言。

第二轮：`diary` 页面 UI 修复。

原因：

- 它是创建流程最关键的输入页。
- 参考图结构清晰，容易按“大日记本 + 照片贴纸 + 检查卡”重排。

第三轮：`character` 页面 UI 修复。

原因：

- 它决定角色一致性概念是否可信。
- 需要补足档案字段和视觉封面，但仍可保持静态 UI。

## 6. 每页允许修改范围

后续逐页修复时，建议每轮只允许修改该页面目录和必要 mock：

- 修 `diary`：只改 `manhua-xcx/pages/diary/*`，必要时改 `manhua-xcx/utils/mock.js`。
- 修 `character`：只改 `manhua-xcx/pages/character/*`，必要时改 `manhua-xcx/utils/mock.js`。
- 修 `generating`：只改 `manhua-xcx/pages/generating/*`，必要时改 `manhua-xcx/utils/mock.js`。
- 修 `chapter-detail`：只改 `manhua-xcx/pages/chapter-detail/*`，必要时改 `manhua-xcx/utils/mock.js`。
- 修 `share`：只改 `manhua-xcx/pages/share/*`，必要时改 `manhua-xcx/utils/mock.js`。
- 修 `mine`：只改 `manhua-xcx/pages/mine/*`，必要时改 `manhua-xcx/utils/mock.js`。
- 修 `quota-empty`：只改 `manhua-xcx/pages/quota-empty/*`，必要时改 `manhua-xcx/utils/mock.js`。
- 如需提取全局样式，必须单独开一轮，只改 `manhua-xcx/app.wxss` 和对应少量页面，不要混在页面修复里。

## 7. 每页禁止修改范围

后续任何 UI 修复轮都应保持：

- 禁止接真实接口。
- 禁止使用 `wx.request`。
- 禁止使用 `wx.uploadFile`。
- 禁止接 AI 服务。
- 禁止真实登录。
- 禁止真实上传图片。
- 禁止创建 Java 后端代码。
- 禁止创建数据库文件。
- 禁止引入依赖。
- 禁止修改 `project.config.json`。
- 禁止扩大 MVP 范围。
- 禁止加入社区、评论、点赞、关注、公开广场。
- 禁止加入支付、充值、会员、订单、余额。
- 禁止展示完整原始日记。
- 禁止修改非本轮页面 UI。
- 禁止自动 push。

## 8. 后续 Codex 执行建议

### 8.1 执行方式

- 不要再批量重做 7 个页面。
- 每轮只修一个页面，最多一个页面 + mock。
- 每轮开始先读取对应 `docs/ui/xx.png` 和现有首页/创建页样式。
- 先写该页面的视觉修复目标，再改代码。
- 修复后必须用 `git diff --name-only` 确认没有动到其他页面。

### 8.2 页面修复模板

每轮建议使用如下任务结构：

1. 当前目标：修复某一个页面静态 UI。
2. 视觉基准：首页、创建页、对应 `docs/ui/xx.png`。
3. 允许修改范围：该页面目录 + 必要 mock + `.ai/TASK.md`。
4. 禁止范围：接口、AI、上传、后端、其他页面。
5. 验证命令：`git status --short`、`git diff --check`、对应页面 JS 的 `node --check`。
6. 自审查：页面是否更接近手账/Q 版/治愈风格，是否没有展示原始日记，是否没有超出 MVP。

### 8.3 设计修复原则

- 先解决结构，再解决颜色。
- 先建立主视觉，再补卡片。
- 先让页面像产品，再让页面像表单。
- 所有开发态“占位”文案从主界面移除，改成 toast 或静态演示说明。
- 页面里最多一个强主按钮，其余为弱按钮或文本按钮。
- 隐私相关页面必须明确“不展示完整原始日记”“仅链接可见”“不公开整本书”。

### 8.4 推荐后续 issues

1. Issue UI-P1：重做章节详情页静态 UI，使其成为漫画阅读页。
2. Issue UI-P2：重做日记输入页静态 UI，使其接近手账日记本和照片贴纸结构。
3. Issue UI-P3：重做角色档案页静态 UI，补足档案字段并强化角色封面。
4. Issue UI-P4：优化生成等待页和分享页，统一状态页表达。
5. Issue UI-P5：优化我的页和免费次数不足页，完成视觉收尾。
