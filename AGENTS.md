# AGENTS.md

## 项目目标

本项目是“日记生成漫画”，目标是开发一个微信小程序：用户每天输入日记并上传照片，系统根据日记、照片、角色档案和情绪标签生成 Q 版漫画章节，最终沉淀为用户自己的私人漫画书。

初版只做个人使用，不做公开社区。

## 技术栈预期

- 微信小程序。
- Java 后端。
- MyBatis。

当前仓库仍处于需求与规则初始化阶段，尚未创建小程序工程、Java 后端工程或数据库结构。

## 固定开发原则

后续所有开发必须遵守：

1. 最小改动。
2. 最小风险。
3. 一次只推进一个小目标。
4. 不整文件重写。
5. 不全文件格式化。
6. 不批量替换。
7. 不做无关重构。
8. 不修改无关业务逻辑。
9. 不修改无关接口字段。
10. 不修改无关数据库结构。
11. 修改后必须自审查。
12. 验证通过后才允许中文 commit。
13. 不自动 push。
14. 如果风险升高，必须停止并说明原因。

## 需求边界

产品定位是私人日记漫画本。初版只服务用户自己记录、查看和私密分享章节。

必须保持以下边界：

- 每个用户初版只有一本默认漫画书。
- 每篇日记生成一个章节。
- 同一天允许创建多章。
- 章节标题由用户手动填写。
- 每章允许上传 1-9 张照片。
- 用户可随机生成页数或自定义 1-8 张。
- 一张生成图表现为多格漫画分镜。
- 初版只提供 Q 版漫画风格。
- 后端采用异步生成任务，小程序轮询任务状态。
- 生成失败后支持免费重试。
- 初版只支持章节私密分享。
- 日记原文属于隐私数据，默认不公开。

## MVP 范围

第一版只做：

- 微信登录。
- 默认漫画书。
- 角色档案。
- 日记章节创建。
- 照片上传记录。
- 异步生成任务。
- Q 版漫画生成。
- 章节查看。
- 章节删除。
- 章节重新生成。
- 章节私密分享。
- 免费次数。
- 简单敏感词过滤。

## 禁止扩大范围

未经用户明确确认，不允许实现或设计以下内容：

- 公开社区、广场、推荐流。
- 评论、点赞、关注。
- 整本书分享。
- 单张漫画图独立分享。
- 支付、订单、充值、会员、余额、支付回调。
- 后台管理系统。
- 复杂内容审核平台。
- 多种漫画风格切换。
- 复杂角色训练。
- 绑定具体 AI 供应商。

## AI 服务规则

- 初版不得绑定具体 AI 图片生成供应商。
- 必须通过可替换 AI 适配层设计文生图、图文生图、任务状态、失败原因、重试和结果图片地址。
- 业务模块不得直接依赖供应商原始字段。
- 漫画整体风格必须保持 Q 版。

## 角色一致性规则

- 用户需要维护固定主角角色档案。
- 角色档案参与每次生成提示词。
- 生成提示词必须要求保持角色形象一致。
- 初版不做复杂角色训练，只做角色档案和提示词约束。
- 如果角色一致性明显失败，允许用户对章节重新生成。

## 隐私与分享规则

- 日记原文属于隐私数据，不公开原文。
- 分享页只展示用户允许分享的章节内容。
- 分享访问者不需要登录，但只能查看私密链接绑定章节。
- 分享页不得暴露其他章节、整本书、用户未授权原文或隐私信息。
- 分享内容不允许被搜索引擎或外部公开索引。

## 每轮开发要求

每一轮 Codex 开发前必须明确：

- 当前小目标。
- 允许修改范围。
- 禁止修改范围。
- 验证命令。
- 是否允许 commit。

每轮只能推进一个小目标。任务不清楚时，应先追问或使用 grill-me，不得直接扩大实现。

## Codex skill 选择规则

每轮开始前，Codex 必须先判断任务类型并明确本轮使用哪个 skill。不得在需求不清楚时直接写代码。

- 新长期项目初始化：使用 setup-matt-pocock-skills。
- 需求不清楚：使用 grill-me。
- 需求整理成 PRD：使用 to-prd。
- PRD 拆分开发任务：使用 to-issues。
- bug / 502 / 404 / 白屏 / 接口异常 / 构建失败：使用 diagnose。
- 支付 / 权限 / 登录 / 注册 / 余额 / 订单 / 回调 / 返利 / 安全修复：使用 tdd。
- 架构审查 / 项目变复杂 / 不确定代码结构是否合理：使用 improve-codebase-architecture 或 zoom-out。
- 长任务交接 / 新对话继续：使用 handoff。

## 后续开发硬性规则

- 每轮只能推进一个小目标。
- 开发前必须确认本轮使用哪个 skill。
- 如果需求不清楚，必须先使用 grill-me，不得直接写代码。
- 如果是安全、支付、权限、登录、余额、订单、回调、返利相关任务，必须优先使用 tdd。
- 如果是 bug 或异常排查，必须优先使用 diagnose。
- 如果任务变大或结构不确定，必须先用 zoom-out 或 improve-codebase-architecture 做只读分析。
- 每轮修改后必须自审查。
- 验证通过后才允许中文 commit。
- 不允许自动 push。
- 风险升高时停止并说明原因。

## 修改后自审查

每轮完成后必须自审查：

- 是否只改了允许范围内的文件。
- 是否引入了非本轮目标内容。
- 是否修改了无关业务逻辑、接口字段或数据库结构。
- 是否引入支付、社区、后台管理等非 MVP 内容。
- 是否存在隐私泄露风险。
- 是否验证通过。

## Commit 规则

- 只有验证通过后才允许 commit。
- commit message 必须使用中文。
- 不允许自动 push。
- 如果当前目录不是 git 仓库，默认不允许自动初始化 git，也不允许 commit；只有用户在本轮明确允许时，才可以初始化 git 并提交。
